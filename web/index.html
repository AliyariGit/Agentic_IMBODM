Protocol Layer

MCP Server

Model Context Protocol (MCP) is Anthropic's open standard for connecting AI assistants to external tools and data. Think of it as USB-C for AI — any MCP client (Claude Desktop, custom agents) can discover and call any MCP server's tools without custom glue code.

How it works
1
FastMCP instance — one line creates the server and registers its name.
2
@mcp.tool() — decorates any Python function as an LLM-callable tool. The docstring becomes the description the LLM reads to decide when to call it.
3
Type hints are the schema — MCP auto-generates the JSON schema from Python types. No manual schema writing needed.
4
@mcp.resource() — exposes read-only context (like rule summaries) the LLM can fetch before deciding what to do.
5
mcp.run() — starts the server. Default transport is stdio, so Claude Desktop spawns it as a subprocess. HTTP transport is also available.
Key Insight
The AI decides when to call the tool. MCP just handles the transport, schema negotiation, and result passing. IBM ODM never knows or cares that an AI called it — it just sees a standard HTTP request.
Orchestration Layer

LangChain Agent

LangChain connects LLMs to tools and orchestrates the reasoning loop. The agent pattern lets GPT-4 read a user message, decide which tool to call and with what arguments, receive the result, and compose a final answer — all in one automated loop.

How it works
1
@tool decorator wraps a Python function. The docstring + type hints become a JSON schema. GPT-4 sees this schema and decides when to invoke it.
2
ChatPromptTemplate defines the agent's identity and task. The agent_scratchpad placeholder is where the LLM's tool calls and their results accumulate.
3
create_openai_tools_agent binds the LLM, tools, and prompt into a single runnable.
4
AgentExecutor runs the loop: LLM → decides to call tool → executes tool → LLM sees result → decides to answer or call another tool.
5
temperature=0 makes extraction deterministic. For payment data, you don't want creative variation in how amounts are parsed.
Key Insight
The agent never hard-codes "extract amount then call ODM". It just sees the user's message and the tool schema, and reasons its way to calling authorize_payment with the right arguments.
Workflow Layer

LangGraph StateGraph

LangGraph turns a LangChain agent into a structured, auditable workflow. Instead of an opaque "agent loop", you define explicit nodes, typed state, and conditional edges — so you can see exactly which path each transaction took.

How it works
1
TypedDict state — every field the workflow reads or writes is declared up-front. Annotated[list, operator.add] means the field is appended to, not replaced, on each node update.
2
Nodes are pure functions — each takes the full state and returns a partial update dict. LangGraph merges the update into state automatically.
3
Conditional edges — a router function reads state and returns a string (the name of the next node). This is how the workflow branches on validation errors or missing fields.
4
compile() validates the graph and returns a Runnable you call with .invoke(initial_state).
5
step_trace accumulates node names as the graph runs, giving a full execution audit trail — which is exactly what the Dashboard panel animates.
Key Insight
Unlike a plain agent loop, the graph is deterministic and inspectable. You can checkpoint it, replay it, add human-in-the-loop nodes, or stream intermediate state — none of which are possible with a black-box agent.
Decision Layer

IBM ODM Integration

IBM Operational Decision Manager is an enterprise Business Rule Management System (BRMS). Rules are authored in near-plain-English BAL inside Decision Center, deployed to the Rule Execution Server, and called via HTDS REST. This is the deterministic, auditable layer the AI cannot replace.

How it works
1
BOM (Business Object Model) — Java domain classes (TransactionRequest, AuthorizationResponse) form the vocabulary rules are written against.
2
BAL rules — written in Decision Center in near-plain English. Business analysts can read and change them without touching code.
3
Ruleflow — rules execute sequentially. The first rule that fires DECLINED short-circuits the rest — same as the mock in this project.
4
HTDS REST endpoint — the @Path("/api/authorize") @POST JAX-RS endpoint accepts JSON, runs the ruleflow via IlrStatelessSession, returns the result.
5
Audit trail — every fired rule is captured in AuthorizationResponse.auditTrail. This is the compliance evidence regulators need.
Key Insight
The AI agent handles ambiguity (what did the user mean?). IBM ODM handles compliance (is this allowed?). Neither can replace the other: the AI can't be audited, and ODM can't read natural language.
